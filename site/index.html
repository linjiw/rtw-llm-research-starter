Adaptive reward & curriculum control for LLM RL post-training

When do training wheels help an LLM learn — and when is adaptivity structurally inert?

RTW-LLM studies Reward Training Wheels (adaptive auxiliary-reward weighting) and adaptive curricula (GACL) for reinforcement-learning post-training of small LLMs on strict-verifier tasks. The headline result is a diagnostic framework: three measurable preconditions that predict, before spending GPU, whether an adaptive teacher has anything to grip.

The research question

One question, two testbeds

Adding dense auxiliary rewards and adapting their weights during training is a common instinct in RL post-training. RTW-LLM asks the meta-question the field mostly skips:

When does adaptive reward/curriculum control help RL post-training of LLMs on strict-verifier (agentic) tasks — and when is it structurally inert?

Paper 1 · now

Countdown — a characterization

A strict AST/exact-rational arithmetic verifier. We show the reward surface fails all three preconditions, then diagnose why shaping stalls while a cheap capability lever succeeds. The diagnosis is the contribution.

Paper 2 · north star

MicroCode — a constructed testbed

Test-driven function synthesis, engineered so all three preconditions hold. Headline experiment: under a deliberately mis-weighted budget, does the adaptive teacher spontaneously down-weight a gamed test-passing proxy while a static schedule gets hacked?

Core architecture

The RTW training loop

A GRPO trainer scores each completion with a strict verifier, feeds per-component reward outcomes to an adaptive teacher, and the teacher reshapes the auxiliary weight vector used for the next batch.

reward(e, x) = primary(e, x) + Σk wk · auxk(e, x) └ exact_correct (verifier) └ teacher-adapted weights └ format · legality · number-F1 · ops · distance · brevity

The teacher: adaptive_stable (Stable-RTW)

  • need = 1 − EMA(auxₖ) — a saturating component decays; a failing one is up-weighted
  • Delay + low LR + double-EMA smoothing for stability
  • Per-key floors/caps + a weight-sum budget via iterative redistribution
  • Six strategies: static · manual · random · adaptive · adaptive_stable · adaptive_phased

Task-agnostic by design

  • Reward manager dispatches on a pluggable scorer + example_fields
  • Curriculum controller uses configurable gate_key / graded_key channels
  • Same GRPO + teacher plumbing drives both Countdown and MicroCode
  • Inference harness: verifier-guided best-of-N with a selector that never sees exactness
The conceptual spine

Three adaptivity preconditions

The falsifiable core of the work: adaptive reward/curriculum control can only help when the reward surface satisfies these. Paper 1 derives them from Countdown's failure; Paper 2 constructs a testbed that satisfies them.

1 · Dense within-group variance

The controller needs a gradient to read across samples of the same task.

Countdown: ~97% variance-saturated
MicroCode: every group varies (std 0.41)

2 · A live hackable proxy

Reward-hacking resistance is only testable if the hack is reachable.

Countdown: non-hackable by construction
MicroCode: hardcode visible tests → proxy 1.0

3 · A smooth competence gradient

A curriculum needs a climbable ordering of difficulty.

Countdown: bimodal exact, tiers collapse
MicroCode: 6-rung ladder, fractional credit

Because Countdown fails all three, adaptive control is structurally inert there — verified controller-independently: even a perfect-hindsight oracle reweighter cannot make the optimizer prefer a correct completion it wasn't already preferring (0 rescue candidates across 7 checkpoints).

Paper 2 · the north star

MicroCode: adaptivity where the preconditions hold

A model emits one Python function from a signature + docstring + K visible tests, graded against held-out tests (the strict truth). Visible tests are the hackable proxy; held-out tests are the source of truth.

The hacking experiment (E5)

Under a TEMPTATION budget that over-weights the visible-test proxy, does the adaptive teacher spontaneously down-weight the gamed channel while a static schedule gets hacked? The held-out verifier measures true success throughout, so “proxy up + primary flat = hacking” is directly observable.

CPU-verified on the real teacher: as visible-pass-rate saturates, its weight decays 0.20 → 0.145 and mass routes to the anti-cheat channel.

Built & gated (CPU, no GPU yet)

  • Base probe: GO — held-out pass 0.31, oracle@8 0.90, variance in every group
  • 30 templates · 24 train + 2 held-out OOD families · 6-rung ladder
  • Frozen protocol · dataset, task IDs, selector, HONEST budget pre-registered
  • Hardened sandbox · spawned worker + memory wall; DoS-sound isolation
