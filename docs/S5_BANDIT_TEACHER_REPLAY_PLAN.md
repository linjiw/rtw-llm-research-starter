## Question (Q3, the strongest reviewer threat)

The RTW-LLM teacher is a lightweight heuristic controller
(delay + double-EMA + floors/caps + weight-sum-budget projection). The
literature already has *learned* multi-reward reweighters for LLM RL — Min et
al. 2024 **DynaOpt** (EXP3 bandit), Lu et al. 2025, Hu et al. 2020 (bilevel).
A reviewer can therefore argue: **the Countdown inertness reflects our weak
controller, not a structural property of the task.** If true, the whole
"adaptivity preconditions" contribution is undermined.

**S5 answers this on CPU, for free, before any GPU spend.**

## What the replay can and cannot show (pre-registered scoping — READ FIRST)

**REVISED after adversarial advisor review (2026-07-20) — three must-fix
amendments folded in; the review computed directly against the logged data.**

DynaOpt's bandit feedback is **counterfactual dev-set improvement under the
newly chosen weights** (EXP3 reward = Σ_i [Mean(R_t,i) − Mean(R_{t−1,i})] on a
dev set, updated every `round_bandit≈10` steps; arms = N+1 = one per reward +
"Do Nothing"). Producing that signal requires **actually training under the
bandit's weights** — a full GPU closed loop. An **offline replay** of logged
`reward_components.jsonl` streams **cannot** manufacture that counterfactual:
the logged component statistics were generated by the policy trained under the
*actual* teacher that produced them (**on-policy circularity** — the stream is
the converged behavior of the very controller we stress-test; the advisor
measured per-key EMA improvement ≈ 0 at steps 240–300, so a naive
improvement-fed bandit is inert *by construction*).

Therefore S5 leads with an **assumption-free per-group ORACLE CEILING**, not a
bandit-behavior test:

> **For each observed GRPO group, is there ANY auxiliary weight vector (perfect
> per-group hindsight) that materially changes the within-group advantage
> structure the optimizer acts on — i.e., flips the top-1 completion or an
> advantage sign — versus uniform weights? If even a perfect-hindsight oracle
> cannot, then NO controller of any strength (bandit, bilevel, policy-gradient)
> can, and Q3 is answered without any feedback-signal approximation.**

The bandit is demoted to a **realism check** (a plausible learned controller
lands where the oracle ceiling says it must), not the load-bearing evidence.

**Correct observable (amendment 1).** GRPO **standardizes advantage magnitude
within-group**, so within-group *variance* is scaled out — the zero-variance
count is the wrong metric (advisor: only 3.17% of groups are zero-variance and
37/38 are dead under *every* weighting → oracle ceiling 0.08pp, an unsatisfiable
threshold that would fabricate Outcome A). What the optimizer actually responds
to is the within-group **ranking / advantage sign**. Observed headroom under
concentrated reweighting: **15.7%** of groups change top-1 — a real, measurable
quantity, unlike the ~0 variance ceiling.

Pre-registered **conclusion ladder** (decided now, not post hoc):

- **Outcome A — even the oracle ceiling is low** (reweighting flips top-1 /
  advantage-sign on a small fraction of groups, and revives ~0 dead groups):
  **assumption-free CPU evidence that no controller can manufacture GRPO signal
  the incumbent missed → precondition failure dominates, not controller weakness.**
  Cite the origin RTW's *Reward-Randomization-is-detrimental* result (movement ≠
  help). **Defuses Q3** without GPU. Claim scope: **"at the observed operating
  point"** — the offline replay cannot speak to a fully different closed-loop
  trajectory, and the doc must say so.
- **Outcome B — the oracle ceiling is high** (reweighting has large ranking
  leverage) AND the DynaOpt bandit exploits a materially different weight
  trajectory: **the controller-strength objection has teeth.** Offline we still
  *cannot* conclude a bandit-driven policy would train better (no
  counterfactual); it **escalates** to a single pre-registered GPU arm
  (bandit-teacher Countdown run, one variable) — a ledger-worthy refinement,
  not a failure.

Honesty guard: this is a **necessary-condition / ceiling probe** ("no controller
can exploit signal that isn't present at this operating point"), explicitly NOT
a closed-loop sufficiency claim. Break the on-policy dependence partially by
also replaying **non-stable streams** (static, v10c2) and the **frozen-weight
delay window** (steps 1–50), where all controllers share uniform weights.

## Design

### Data (frozen, already committed)
Logged `outputs/checkpoints/grpo_{static,stable,v10c2,v12,v13sft}_*/reward_components.jsonl`
— 4800 rows per 300-step run; 16 completions per `reward_batch_index` (one
optimizer step); each row carries the full `components` dict (all six aux keys +
`correct`). Reconstruct groups of `num_generations` (=4) consecutive rows within
each `reward_batch_index`, matching `rewards.py`'s positional group layout. The
primary comparison stream is **stable seed 0/1/2** (the run whose teacher we are
stress-testing); static/v10c2/v12/v13 available as secondary streams.

### Arms compared (all fed the identical observed group stream)
1. **ORACLE ceiling (primary evidence, no new strategy — pure analysis).** For
   each group, evaluate uniform 0.2 weights and the extreme reweightings (the
   six one-hot-concentrated aux vectors + primary-only), compute the per-group
   **ranking-leverage** observable below, and take the per-group *max* achievable
   change. This is the assumption-free ceiling: perfect per-group hindsight.
2. **`adaptive_stable`** — the incumbent, re-run offline on the stream.
   **Sanity/correctness guard:** its offline weight trajectory must match the
   committed `teacher_weights.jsonl` from that run to within EMA/float tolerance
   (confirmed reproducible: `teacher.update` depends only on batch-averaged
   components, `rewards.py:133`, `teacher.py:162-168`).
3. **`bandit` (new strategy, DynaOpt-faithful, default-off — realism check).**
   EXP3 over N+1 arms (six aux keys + "Do Nothing"); a chosen arm increments
   that key's weight; update every `round_bandit` steps. Faithful to Min et al.
   2024 (EXP3, N+1 arms, periodic update). **Arm-reward (amendment 3):** the
   *available within-group ranking leverage* the arm would create at step t
   (how much concentrating on key i changes top-1/advantage-sign vs the current
   weights), NOT already-exhausted EMA improvement (which is ≈0 at stable
   convergence and would bake in inertness). The dev-set-feedback substitution
   is disclosed; the bandit is secondary color, not the load-bearing claim.

### Metrics (per group, per stream)
- **PRIMARY — ranking leverage (amendment 1):** fraction of groups where the
  within-group **top-1 completion** changes under a reweighting vs uniform;
  fraction where any per-completion **GRPO advantage sign** flips; and — most
  pointed — fraction where a *correct* completion's advantage sign flips (does
  reweighting ever help the optimizer prefer a correct completion it otherwise
  wouldn't?). Report the oracle *ceiling* and each controller's *realized* value.
- **Dead-group revival (reported, not thresholded):** fraction of zero-variance
  groups any reweighting can revive (advisor ceiling ≈0.08pp) — reported as
  context for why variance is the wrong lever, not as a decision criterion.
- **Weight movement (secondary):** bandit/adaptive_stable final-vs-init L1, max
  per-step L1 — vs adaptive_stable's known near-uniform endpoint (L1 ~0.16 off
  uniform, cross-seed std ~0.003).

### Decision rule (pre-registered)
Scored against the **multi-seed** stable stream (never one seed), plus the
static/v10c2 streams and the delay-window slice to break on-policy dependence.
- **Outcome A** if the **oracle top-1-flip ceiling is low** (< ~15% of groups)
  AND the *correct*-completion advantage-sign-flip ceiling is negligible
  (< ~2% of groups) — i.e., even perfect hindsight barely changes what the
  optimizer prefers, and almost never rescues a correct completion.
- **Outcome B** if the oracle ceiling is high AND the DynaOpt bandit *realizes*
  a materially different weight trajectory (final-vs-init L1 **> 2×**
  adaptive_stable's) — then escalate to the single pre-registered GPU arm.
Thresholds are provisional and will be locked to the observed
uniform-baseline distribution in the script header before the run is scored;
the *shape* of the rule (oracle-ceiling-first, correct-flip as the sharp test)
is fixed now.

## Implementation constraints (IMPLEMENT stage, after ADVISE — advisor amendments folded)
- **Oracle ceiling is pure analysis** in the replay script (no strategy needed)
  — this is the primary, assumption-free evidence and should be built first.
- One variable for the realism check: add a `bandit` strategy to `teacher.py`
  (`VALID_STRATEGIES` + an `_bandit_update`), default-off; **Countdown defaults
  byte-identical** (existing strategies untouched; a new-strategy branch only).
- The replay is a new additive analysis script (`scripts/19_bandit_replay.py`),
  reads only committed JSONL, writes `outputs/bandit_replay.json`. No frozen
  component touched. No GPU.
- **Group reconstruction (verified SOUND by advisor):** groups = 4 consecutive
  rows (`num_generations`) within each `reward_batch_index` (16 rows/batch, 300
  batches, layout `[g0×4 | g1×4 | ...]`). Assert group_size divides rows/batch
  **and** that the 4 rows in each reconstructed group share one `id` (guards
  against a std-based grouping shortcut that coincidentally merges groups).
- Unit tests: EXP3 arm-weight update math; "Do Nothing" arm; determinism under
  fixed seed; the adaptive_stable offline-replay-matches-logged-trajectory
  sanity check; the group-id assertion; the ranking-leverage/top-1-flip metric
  on a hand-built group with a known flip.
- Ruff clean; full suite green.

## Risks / failure modes to watch
- **Offline-feedback approximation is not DynaOpt's true signal** — disclosed
  above; the doc must not claim "we ran DynaOpt on Countdown," only "a
  DynaOpt-faithful EXP3 controller replayed on observed component streams."
- **Replay-harness bug** silently changing the stream → the adaptive_stable
  trajectory-match sanity check is the guard; if it fails, stop.
- **Over-reading Outcome B** → pre-committed to escalating, not concluding.
- **Group reconstruction mismatch** with `rewards.py`'s positional layout →
  assert group size divides row count per `reward_batch_index`.

## Not in scope (kill-list respected)
No GPU this step; no closed-loop bandit training unless Outcome B triggers the
single pre-registered escalation arm; no change to the verifier, selector,
